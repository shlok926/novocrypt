FEATURE 2: CRYPTOGRAPHY SCANNER
--------------------------------------------------------------------------------
PAGE ROUTE  : /scanner

DESCRIPTION:
  User pastes a URL or code snippet. Tool scans for vulnerable cryptographic
  implementations and shows quantum risk assessment.

UI COMPONENTS:
  - ScannerTabs.tsx         : Switch between URL / Code / API Key scan
  - URLScanForm.tsx         : URL input + scan button
  - CodeScanForm.tsx        : Monaco editor for code paste
  - ScanResults.tsx         : Vulnerability findings list
  - ScanVulnerabilityCard.tsx : Per-finding detail card
  - ScanSummaryBadge.tsx    : Overall risk badge
  - ScanHistoryList.tsx     : Past scans list

BACKEND:
  - POST /api/scanner/url           : Scan a website URL
  - POST /api/scanner/code          : Scan pasted code snippet
  - GET  /api/scanner/history       : User scan history
  - GET  /api/scanner/result/:id    : Get specific scan result

URL SCAN LOGIC:
  1. Fetch SSL/TLS certificate from URL (via ssl-checker npm)
  2. Extract: key algorithm, key size, cipher suite, expiry
  3. Check: RSA < 4096? → Vulnerable. ECC P-256? → Moderate.
  4. Check: TLS version (1.0/1.1 → Critical, 1.2 → Moderate, 1.3 → Better)
  5. Return: vulnerability list with severity scores

CODE SCAN LOGIC:
  1. Regex pattern matching for known vulnerable patterns:
     - crypto.createCipher('des', ...) → Critical
     - RSA key size < 2048 → High
     - MD5/SHA1 usage → High
     - Math.random() for crypto → Critical
     - Hardcoded keys → Critical
  2. Language detection (JS/TS/Python/Java/Go)
  3. AST-based analysis for deeper detection
  4. Return: line numbers + vulnerability descriptions

DATA MODEL:
  TABLE: scan_results
    id              UUID PRIMARY KEY
    user_id         UUID REFERENCES users(id)
    scan_type       VARCHAR(50)   -- 'url' | 'code' | 'api'
    input_target    TEXT
    findings        JSONB         -- array of vulnerability objects
    overall_score   INTEGER       -- 0-100
    risk_level      VARCHAR(50)
    scanned_at      TIMESTAMP DEFAULT NOW()

FINDINGS JSON STRUCTURE:
  {
    "id": "uuid",
    "type": "RSA_WEAK_KEY",
    "severity": "high",
    "description": "RSA-1024 found, vulnerable to quantum attack",
    "location": "line 42 / ssl-cert",
    "recommendation": "Upgrade to RSA-4096 or migrate to CRYSTALS-Kyber",
    "quantum_breakable": true,
    "estimated_break_time": "2-5 years with quantum computer"
  }

NPM PACKAGES:
  - ssl-checker          : SSL certificate analysis
  - node-forge           : Crypto analysis
  - @babel/parser        : JS/TS AST analysis

FEATURE 5: ALGORITHM PLAYGROUND
--------------------------------------------------------------------------------
PAGE ROUTE  : /playground

DESCRIPTION:
  Live browser-based tool where users can type a message, encrypt it with RSA,
  then simulate a quantum attack to decrypt it. Visual step-by-step breakdown.

UI COMPONENTS:
  - PlaygroundLayout.tsx        : Split-screen layout
  - MessageInput.tsx            : Text area for message input
  - EncryptionPanel.tsx         : Encrypt with RSA / Kyber
  - DecryptionPanel.tsx         : Classical vs Quantum decrypt
  - AttackVisualization.tsx     : Animated attack breakdown
  - BitBreakdown.tsx            : Visual bit-by-bit analysis
  - AlgorithmSelector.tsx       : Choose RSA-512/1024/2048 or Kyber
  - SpeedComparison.tsx         : Real-time speed comparison bars
  - StepByStepLog.tsx           : Console-style step log

BACKEND:
  - POST /api/playground/encrypt         : Encrypt message
  - POST /api/playground/decrypt         : Decrypt ciphertext
  - POST /api/playground/quantum-attack  : Simulate quantum attack
  - GET  /api/playground/algorithms      : Available algorithms list

PLAYGROUND FLOW:
  1. User types: "Hello, my bank password is X123"
  2. Selects: RSA-512 encryption
  3. Clicks Encrypt → Shows: ciphertext hex string
  4. Classical attack: Shows "Est. time: 2.3 million years"
  5. Quantum attack (Shor's): Shows step-by-step:
     - Step 1: Choose random a = 7
     - Step 2: Find period r of a^x mod n
     - Step 3: Compute GCD(a^r/2 ± 1, n)
     - Step 4: Factors found: p=..., q=...
     - Step 5: Compute phi(n), then d
     - DECRYPTED: "Hello, my bank password is X123"
  6. Then shows Kyber encryption → "Quantum-resistant!"

SUPPORTED ALGORITHMS:
  Classical (Vulnerable):
    - RSA-512   (Demo only, very fast to break)
    - RSA-1024  (Fast quantum break)
    - RSA-2048  (Moderate quantum break)

  Post-Quantum (Safe):
    - CRYSTALS-Kyber-512
    - CRYSTALS-Kyber-768
    - CRYSTALS-Kyber-1024

NPM PACKAGES:
  - node-forge       : RSA operations in browser
  - crypto (built-in): For demonstrations

FEATURE 6: COMPLIANCE CHECKER
--------------------------------------------------------------------------------
PAGE ROUTE  : /compliance

DESCRIPTION:
  User selects their industry and inputs their current encryption setup.
  Tool checks compliance against major standards and generates gap report.

UI COMPONENTS:
  - ComplianceForm.tsx          : Industry + encryption input form
  - ComplianceResults.tsx       : Results grid
  - ComplianceStandardCard.tsx  : Per-standard result (Pass/Fail/Warning)
  - GapAnalysisTable.tsx        : What's missing vs required
  - ComplianceReport.tsx        : Full downloadable report
  - RemediationSteps.tsx        : How to fix each gap

BACKEND:
  - POST /api/compliance/check          : Run compliance check
  - GET  /api/compliance/standards      : List all standards
  - GET  /api/compliance/standard/:id   : Standard details
  - GET  /api/compliance/history        : User check history
  - POST /api/compliance/export         : Export as PDF

SUPPORTED STANDARDS:
  - NIST SP 800-208      : Quantum-safe cryptography guidelines
  - NIST FIPS 203        : CRYSTALS-Kyber standard
  - NIST FIPS 204        : CRYSTALS-Dilithium standard
  - HIPAA                : Healthcare data protection
  - PCI-DSS v4.0         : Payment card industry
  - GDPR                 : EU data protection
  - SOC 2 Type II        : Service organization controls
  - ISO/IEC 27001:2022   : Information security management

COMPLIANCE CHECK LOGIC:
  For each standard, check:
    1. Key size requirements met?
    2. Approved algorithms used?
    3. TLS version acceptable?
    4. Key rotation policy?
    5. Quantum-safe roadmap required?

  Result per standard:
    PASS    : All requirements met
    WARNING : Some requirements met, action needed
    FAIL    : Critical requirements not met
    N/A     : Standard not applicable to industry

DATA MODEL:
  TABLE: compliance_checks
    id              UUID PRIMARY KEY
    user_id         UUID REFERENCES users(id)
    industry        VARCHAR(100)
    encryption_data JSONB
    results         JSONB
    overall_status  VARCHAR(50)
    checked_at      TIMESTAMP DEFAULT NOW()

  TABLE: compliance_standards
    id              UUID PRIMARY KEY
    code            VARCHAR(50)   -- 'NIST-800-208', 'HIPAA', etc.
    name            VARCHAR(200)
    description     TEXT
    requirements    JSONB
    applicable_to   JSONB         -- array of industries
